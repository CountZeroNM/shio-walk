# Debug Notes / 引き継ぎ用デバッグノート

本ドキュメントは **shio-walk プロジェクトにおける長期デバッグの記録と設計判断の背景** をまとめたものです。  
README には書ききれない「考え方」「学び」「再発防止策」を中心に記載します。

---

## 1. 本プロジェクトの前提思想（超重要）

### Single Source of Truth

本プロジェクトでは **バックエンド API を唯一の信頼できる情報源（Single Source of Truth）** とします。

- フロントエンドでのデータ補正・吸収は原則禁止
- 不整合があった場合は **必ず API 側を修正**
- フロントエンドは「受け取ったデータをそのまま描画する」責務に限定する

この方針を守らない場合、以下の問題が発生します：

- re-frame の state が壊れる
- UI が一瞬遷移して元に戻る
- デバッグが指数関数的に困難になる

---

## 2. 今回の不具合概要（要約）

### 症状

- ウォーキング開始ボタンを押す
- 一瞬「ウォーキング中UI」に遷移
- すぐに元の画面に戻る
- API は成功レスポンスを返している
- エラーは出ない、もしくは warning のみ

### 結論

**単一のバグではなく、複数の不整合が連鎖した状態不整合（レースコンディション）**

---

## 3. 実際に起きていた問題

### API レスポンスの揺れ

- `status`
  - `"active"` と `"in_progress"` が混在
- key 命名
  - `snake_case` と `kebab-case` が混在
- 数値フィールド
  - `steps`, `distance-meters` が `nil` になるケース
- 一部データで
  - `id` が `nil`

### フロントエンド側の状態

- optimistic update により UI が先に切り替わる
- 直後に `load-walks` で API データが流入
- 不整合なデータで state が上書きされ UI が巻き戻る

---

## 4. 設計判断（なぜこの修正を選んだか）

### 選択肢

- ❌ フロントエンドで吸収する
- ⭕ API 契約を正す

### 判断理由

- フロントで吸収すると：
  - 不具合が隠蔽される
  - 別画面・別開発者で再発する
  - 将来のデバッグコストが爆増する
- API 契約を正すと：
  - re-frame の状態管理が単純になる
  - UI 巻き戻りが構造的に起きなくなる
  - 引き継ぎが容易になる

**短期的コストより、長期的安全性を優先**

---

## 5. コーディング上の重要な学び（核心）

### 5.1 re-frame の UI 巻き戻りは「ほぼ state 不整合」

以下の症状が揃ったら、まず疑う：

- 一瞬だけ UI が切り替わる
- API は成功している
- エラーが出ない

👉 **state が二重に更新されている可能性が極めて高い**

---

### 5.2 optimistic update は強力だが危険

optimistic update とは：

> API 成功を仮定して UI を先に更新する手法

前提条件：

- API レスポンスが **完全に安定していること**
- status / key / 値が揺れないこと

これが守られないと、**UI 巻き戻り製造機** になる。

---

### 5.3 「フロントで吸収」は将来の自分を殺す

書きがちなコード：

```clojure
(or (:distance-meters walk) 0)


これは：

API の破綻を隠す

問題を別箇所へ拡散する

後続開発者を混乱させる

👉 正解は API 側で必ず値を保証する

## 6. 再発防止チェックリスト（実務用）
### API 側チェック

- key はすべて kebab-case

- id は必ず存在する

- 数値フィールドは nil を返さない

- status は固定値のみ（例: "active", "completed"）

- start/update/complete でレスポンス構造が一致している

### フロントエンド側チェック（re-frame）

- 同じ event-id を複数回 reg-event-* していない

- optimistic update と API 更新が競合していない

- load-* 系イベントが state を意図せず上書きしていない

- current-* 系 state が API データで壊されていない

---

デバッグ時の確認順序（推奨）

API の raw response を console に出す

key / status / nil の有無を確認

re-frame の event 発火順を追う

「どの event が最後に state を書き換えたか」を特定

フロントを直す前に API 契約を疑う

## 修正履歴（2026-03-06）

### ハイフンとアンダースコアの不整合による深刻な不具合

- **不具合症状**:
  - 完了ボタン押下で「完了失敗」エラーが発生し、履歴が「進行中」から変わらない。
  - 更新ボタン押下で入力した距離が 0.00km に戻ってしまう。
  - エラーの閉じるボタンが反応しない。

- **根本原因**:
  - バックエンド API ハンドラにおけるキー名参照ミス。
  - DB から返ってくる実際のキーは `:walks/distance_meters` (アンダースコア) だが、ハンドラが `:walks/distance-meters` (ハイフン) で `nil` を取得していた。
  - ユーザー ID についても同様のミスがあり、`"nil"` 文字列を UUID として DB に渡したことで PostgreSQL がパースエラー（500）を出していた。
  - フロントエンドに `:clear-error` ハンドラが欠落していた。

- **解決策**:
  - ハンドラ層で DB から取得する際は `snake_case` (アンダースコア) を使い、API レスポンス（JSON）として返却する際に `kebab-case` (ハイフン) に変換する。
  - フロントエンドにエラー消去イベントを実装。

- **参照先**:
  - `GEMINI.md`: 「4. 内部データと外部データのキー変換」
  - `README.md`: 「開発上の注意：キー命名規則」
  - `backend/src/shio_walk/walks/handlers.clj`: 修正済みコード

## 修正履歴（2026-03-06 追記）

### 統計情報の「獲得報酬」が 0 のまま更新されない不具合

- **不具合症状**:
  - 報酬パネルでは「🎉」が表示されている（アンロック済み）にもかかわらず、統計情報の「獲得報酬」カウントが 0 のまま。

- **根本原因**:
  1. **ロジックの漏れ (DB層)**: `check-and-unlock-rewards!` が `user_rewards` テーブルにレコードを挿入するだけで、`user_stats` テーブルの `rewards_unlocked` カラムを更新（同期）していなかった。
  2. **キー名参照ミス (Handler層)**: `get-stats` ハンドラが、DB から取得した `:user_stats/rewards_unlocked` を `:user_stats/rewards-unlocked` (ハイフン) で参照していたため、常に `nil` が返っていた。

- **解決策**:
  - `check-and-unlock-rewards!` 実行時に、`user_rewards` の総数をカウントして `user_stats.rewards_unlocked` を `UPDATE` する処理を追加。
  - `get-stats` ハンドラのキー参照をアンダースコアに修正。

- **教訓**:
  - 複数のテーブルにまたがる集計値（デノーマライズされた値）を扱う際は、更新タイミング（トリガー）を明確にする。
  - `get-stats` のような集計 API は、常に DB の実際のカラム名（アンダースコア）と API 規約（ハイフン）の境界線である。

## 修正履歴（2026-03-08）

### iOS Safari におけるセンサー起動およびデータ同期の信頼性向上

- **不具合症状**:
  - iPhone の Safari で「センサー計測を開始」をタップしても、許可ダイアログが表示されず歩数がカウントされない。
  - バックグラウンドで歩いている際、定期的なデータ同期が走らず、アプリに戻った時に古い数値が表示される。
  - 手動入力した数値が、完了ボタン押下時に 0 に上書きされてしまう。

- **根本原因**:
  1. **iOS User Gesture 制約**: `DeviceMotionEvent.requestPermission` は、ユーザーのクリックイベントに直結した同期的なコールスタック内でのみ許可される。re-frame の `event-fx` を介した非同期呼び出しでは拒否されていた。
  2. **バックグラウンド制限**: iOS Safari はバックグラウンドでの `setInterval` を抑制するため、5分おきのタイマーが期待通りに動作していなかった。
  3. **データ参照の不整合**: `complete-walk` イベントが `:sensor` バッファ（手動入力時は 0）を優先して参照していたため、`:current-walk` への手動入力内容が無視されていた。

- **解決策**:
  - **直接呼び出し**: `dashboard.cljs` のボタン `on-click` ハンドラから直接 `sensors/start-sensors!` を呼び出す構成に変更。
  - **Foreground 同期**: `VisibilityChange` イベントを利用し、ユーザーがアプリ画面に戻った瞬間に強制的に同期を実行。
  - **参照の統一**: 同期・完了時のデータソースを `:current-walk` に一本化し、手動・センサー両方の入力を許容。

- **UI/UX 改善**:
  - 統計パネルを 2列グリッド化し、モバイルでの視認性を向上。
  - ウォーク履歴を最新3件に制限し、折りたたみ表示（アコーディオン）を実装。
  - 手動入力欄にローカル `atom` を導入し、入力中の挙動を安定化。

- **教訓**:
  - モバイルブラウザ（特に iOS）のセキュリティ制約とバックグラウンド動作の制限を前提とした設計が必要。
  - re-frame のグローバル状態と UI コンポーネントのローカル状態（入力中の一時データ）の責務を明確に分ける。
